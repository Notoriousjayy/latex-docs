\chapter*{Curriculum Title and Purpose}
\addcontentsline{toc}{chapter}{Curriculum Title and Purpose}

\section*{1. Curriculum Title}
\textbf{\large Hands-On Web Application Security: A Foundations-to-Advanced Curriculum} \\
\textit{Bridging WEB-200 (OSWA) and WEB-300 (OSWE) for authorized, lab-based offensive and defensive practice.}

\section*{2. Curriculum Purpose}
This curriculum prepares learners to \emph{find}, \emph{validate}, \emph{communicate}, and \emph{remediate} web application vulnerabilities through disciplined, hands-on practice in authorized lab environments. It is engineered as a single, continuous learning arc that begins with HTTP fundamentals, the assessor's toolchain, and the foundational vulnerability classes drawn from \textbf{WEB-200 (OSWA)} --- including XSS, CSRF, CORS misconfigurations, SQL injection, directory traversal, XXE, SSTI, SSRF, command injection, and IDOR --- and progresses into the source-code-assisted exploitation patterns drawn from \textbf{WEB-300 (OSWE)} --- including JavaScript prototype pollution, advanced SSRF, persistent and DOM-based XSS, session hijacking, weak token generation, .NET deserialization, blind SQLi, PostgreSQL extension and large-object abuse, UDF reverse shells, magic hashes and PHP type juggling, file-upload, regex and character-restriction bypasses, RCE via database functions, XXE for data exfiltration, and OS command injection over WebSockets.

The curriculum bridges the two syllabi by inserting a deliberate \emph{exploitation-validation, evidence, and source-code-analysis} bridge (Modules~10--11) so that learners do not merely \emph{run exploits} but learn the repeatable, methodology-driven discovery process required by mature AppSec programs. Every module pairs offensive understanding with a defensive validation or remediation exercise so the curriculum is equally usable for Red Team uplift, Blue Team verification, and secure-coding training.

\section*{3. Target Learners}

\subsection*{3.1 Primary Learner Profile}
The curriculum is designed for practitioners who already write or operate code and want to become competent web application assessors:
\begin{itemize}
  \item Junior-to-mid penetration testers and AppSec engineers preparing for OSWA / OSWE.
  \item DevSecOps and platform engineers who own authentication, authorization, or input handling.
  \item Software engineers transitioning into security engineering roles.
  \item Bug-bounty hunters who want a methodology rather than pattern-matching.
  \item Defenders (SOC, IR, vulnerability management) who must \emph{verify} reported web application issues.
\end{itemize}

\subsection*{3.2 Prerequisites and Assumed Skills}
\begin{itemize}
  \item Comfort with Linux command-line basics and a scripting language (Python, JavaScript, or Bash).
  \item Working understanding of HTTP, HTML, and the request/response model.
  \item Familiarity with at least one of: relational databases, web frameworks, or browser fundamentals.
  \item Ability to read --- not necessarily write --- code in two or more of: PHP, Java, .NET (C\#), Node.js, Python.
\end{itemize}

\subsection*{3.3 Readiness Levels and Accommodations}
The curriculum recognizes three readiness levels and provides scaffolding for each:
\begin{itemize}
  \item \textbf{Foundation track:} Learners new to security receive expanded ``Module Primer'' readings, paired-lab partners, and a \emph{guided} exploitation path.
  \item \textbf{Standard track:} Learners with some scripting and HTTP literacy follow the default sequence.
  \item \textbf{Accelerated track:} Experienced engineers may compress Modules 1--3 into a single diagnostic week and devote freed capacity to the WEB-300-aligned modules and capstone.
\end{itemize}

\chapter*{Curriculum Structure and Delivery}
\addcontentsline{toc}{chapter}{Curriculum Structure and Delivery}

\section*{4. Curriculum Structure}
The curriculum is organized into \textbf{six phases} and \textbf{sixteen modules}. The phases enforce a logical scope and sequence, moving from orientation to advanced exploitation and finally to professional reporting and remediation.

\begin{tcolorbox}[callout,colback=afpaper,colframe=afnavy,colbacktitle=afnavy,title=Phase Map]
\begin{description}[leftmargin=2.4cm,style=nextline]
  \item[Phase A] \textit{Orientation, Ethics, and Tooling.} Module 1.
  \item[Phase B] \textit{HTTP, Web Fundamentals, and Assessment Workflow.} Modules 2--3.
  \item[Phase C] \textit{Foundational Vulnerability Classes.} Modules 4--9.
  \item[Phase D] \textit{Exploitation Validation, Evidence, and Source-Code Analysis.} Modules 10--11.
  \item[Phase E] \textit{Advanced Exploitation Patterns.} Modules 12--15.
  \item[Phase F] \textit{Reporting, Remediation, and Capstone.} Module 16.
\end{description}
\end{tcolorbox}

\subsection*{4.1 Pacing for Different Formats}
\begin{itemize}
  \item \textbf{12-week semester (default):} 1 module / week with capstone spanning weeks 15--16.
  \item \textbf{8-week intensive:} Modules 1--3 compressed into Week 1; Modules 4--6 paired into Weeks 2--3; capstone runs Weeks 7--8.
  \item \textbf{Bootcamp (4 weeks, 40~hrs/wk):} Phases A--C in Week 1; Phase D mid-Week 2; Phase E Weeks 2--3; capstone Week 4.
  \item \textbf{Self-paced cohort:} Weekly synchronous lab review, asynchronous reading and exploit work, capstone after Module 15.
\end{itemize}

\begin{ethics}
\textbf{Authorized lab environments only.} Every lab in this curriculum is performed against intentionally vulnerable applications, instructor-provided container stacks, or contracted training platforms. Learners must read and sign a \emph{Lab Code of Conduct} before Module 1, Lab 1. Attacking systems the learner does not own or does not have explicit written authorization to test is a violation that results in immediate removal from the program and may constitute a criminal offense under jurisdictional computer-misuse laws.
\end{ethics}

\chapter{Orientation, Ethics, and the Authorized Lab}
\textbf{Phase A --- Orientation, Ethics, and Tooling}

\subsection*{Topics Covered}
\begin{itemize}
  \item Authorization, scoping, and the difference between testing and exploitation.
  \item Rules of engagement, statements of work, and lab-environment authorization letters.
  \item Evidence handling: timestamps, hashes, screenshot hygiene, chain of custody.
  \item Setting up a hardened tester workstation and isolated lab network.
  \item Professional communication, responsible disclosure, and reporting expectations.
\end{itemize}

\begin{outcomes}
By the end of Module 1, learners will be able to:
\begin{enumerate}
  \item \textbf{Identify} the elements of a written authorization that legally permit a web application assessment.
  \item \textbf{Demonstrate} a documented evidence-handling workflow in their lab notebook.
  \item \textbf{Explain} how the curriculum's lab environment differs from production targets and why that distinction matters legally and ethically.
  \item \textbf{Configure} an isolated lab workstation (host, VM, or container) with the curriculum's baseline toolchain.
\end{enumerate}
\end{outcomes}

\begin{labbox}[Lab 1.1: Lab Charter and Range Stand-Up]
\textbf{Scenario.} You have just been onboarded to an AppSec team. Before any testing begins, you must produce a written \emph{Lab Charter} that formalizes how, where, and against what you will practice.\\[3pt]
\textbf{Authorized lab environment.} A locally provisioned VM or Docker stack hosting OWASP Juice Shop, DVWA, and Mutillidae II. No internet egress to non-authorized targets.\\[3pt]
\textbf{Tools used.} Burp Suite Community or OWASP ZAP, browser DevTools, Git, a Markdown editor, and an evidence directory layout supplied by the instructor.\\[3pt]
\textbf{Student task.} (1) Draft a one-page Lab Charter declaring the authorized targets, prohibited actions, evidence layout, and the learner's signed acknowledgement. (2) Stand up the local container stack and verify reachability. (3) Capture three labeled screenshots demonstrating the running stack and proxy interception.\\[3pt]
\textbf{Step boundaries.} No traffic is to leave the host. No real-world targets. No payload execution outside the provided containers.\\[3pt]
\textbf{Expected evidence.} Signed Lab Charter (PDF); \tool{docker ps} output; three timestamped screenshots; SHA-256 of the evidence archive.\\[3pt]
\textbf{Reflection question.} What is the smallest change you could make to your workflow so that, six months from now, an instructor reading your evidence could reproduce this lab without contacting you?\\[3pt]
\textbf{Defensive / remediation component.} List three policy controls (technical or administrative) that, if applied to a production tester's workstation, would make accidental out-of-scope testing measurably less likely.\\[3pt]
\textbf{Success criteria.} Charter is complete and signed; lab stack is running; evidence archive is hashed; reflection demonstrates a concrete operational improvement.
\end{labbox}

% --------------------- MODULE 2 ---------------------
\chapter{HTTP, Web Architecture, and the Browser Trust Model}
\textbf{Phase B --- HTTP, Web Fundamentals, and Assessment Workflow}

\subsection*{Topics Covered}
\begin{itemize}
  \item HTTP/1.1 and HTTP/2 message structure; methods, status codes, and conditional requests.
  \item TLS handshake essentials and what the proxy can and cannot see.
  \item Cookies, \tool{SameSite}, \tool{HttpOnly}, \tool{Secure}, and the same-origin policy.
  \item Content Security Policy as a read signal during assessment.
  \item Common architectural shapes: SPA + JSON API, classical server-rendered, BFF.
\end{itemize}

\begin{outcomes}
By the end of Module 2, learners will be able to:
\begin{enumerate}
  \item \textbf{Analyze} an HTTP request/response pair and label every header's purpose and security relevance.
  \item \textbf{Compare} the same-origin policy with CORS and articulate why CORS exists.
  \item \textbf{Demonstrate} interception, modification, and replay of an HTTP request through an authorized proxy.
  \item \textbf{Identify} from headers alone the likely framework or platform behind a target lab application.
\end{enumerate}
\end{outcomes}

\begin{labbox}[Lab 2.1: HTTP Anatomy and Tampering Workshop]
\textbf{Scenario.} A new training portal is about to ship. You must produce an annotated HTTP trace that the development team can use to harden response headers.\\[3pt]
\textbf{Authorized lab environment.} OWASP Juice Shop running locally; intercepting proxy of choice.\\[3pt]
\textbf{Tools used.} Burp Suite or ZAP; browser DevTools; \tool{curl}; \tool{httpie}.\\[3pt]
\textbf{Student task.} Capture five distinct request/response pairs covering: anonymous browse, login, an authenticated state-changing action, a JSON API call, and a static asset. For each, annotate every meaningful header and identify two strengths and two weaknesses.\\[3pt]
\textbf{Step boundaries.} No exploitation attempts; observation, replay, and header rewriting only.\\[3pt]
\textbf{Expected evidence.} Annotated trace document (Markdown or PDF), exported \tool{.har} file, and a one-page header hardening recommendation.\\[3pt]
\textbf{Reflection question.} Which header surprised you most, and how would you explain its importance in 30 seconds to a developer who has never set it?\\[3pt]
\textbf{Defensive / remediation component.} Author a minimal but production-realistic set of response-header recommendations (CSP, HSTS, COOP/COEP, cookie flags) for the lab application.\\[3pt]
\textbf{Success criteria.} All five traces captured and annotated; hardening recommendation is implementable and references vendor documentation.
\end{labbox}

% --------------------- MODULE 3 ---------------------
\chapter{Web Assessment Methodology and the Tester's Toolchain}

\subsection*{Topics Covered}
\begin{itemize}
  \item A repeatable assessment workflow: scope $\to$ map $\to$ enumerate $\to$ probe $\to$ validate $\to$ report.
  \item Mapping: site crawl, JS endpoint extraction, parameter enumeration.
  \item Fuzzing with \tool{ffuf} and templated scanning with \tool{nuclei} (lab targets).
  \item Note-taking patterns and reproducible evidence collection.
\end{itemize}

\begin{outcomes}
By the end of Module 3, learners will be able to:
\begin{enumerate}
  \item \textbf{Construct} a written assessment methodology checklist appropriate for a small web application.
  \item \textbf{Demonstrate} parameter and content discovery using \tool{ffuf} against an authorized lab target.
  \item \textbf{Explain} the trade-offs between manual proxy-driven and templated automated scanning.
  \item \textbf{Document} findings in a way that another tester can reproduce.
\end{enumerate}

\subsection*{Formative Checks}
Each module ends with a 10-question, low-stakes check that prioritizes \emph{reasoning}. Sample item: ``Given a CORS response with \tool{Access-Control-Allow-Origin} reflected from \tool{Origin} and \tool{Access-Control-Allow-Credentials: true}, classify the risk and justify.''

\subsection*{Practical Demonstrations}
Twice during the curriculum (after Modules 6 and 12), each learner performs a 30-minute live walkthrough of one of their lab findings against an unfamiliar but equivalent lab target. The instructor evaluates against a rubric covering methodology, evidence quality, and communication.

\subsection*{Reflection Journal}
Each module's reflection question (see lab specifications) becomes a journal entry. Entries are graded for depth and growth, not for ``correctness.''

\subsection*{Secure Remediation Exercises}
Each lab includes a defensive component. These are graded together at mid-term and end-term as a single ``defensive portfolio.''

\chapter*{Capstone Project}
\addcontentsline{toc}{chapter}{Capstone Project}

\section*{9. Capstone Project}
The capstone is an authorized end-to-end web application assessment performed against an instructor-supplied lab application that the learner has not previously encountered.

\subsection*{9.1 Scenario}
``BlueRiver Health Systems'' is a fictitious organization that has commissioned a pre-launch web application assessment of its new patient-portal. The instructor delivers: (1) a written authorization letter, (2) a target lab stack, (3) a stub of internal system documentation, and (4) a sample executive summary from a prior engagement.

\subsection*{9.2 Rules of Engagement}
\begin{itemize}
  \item Targets are limited to the supplied lab stack. No traffic to non-authorized hosts.
  \item Denial-of-service and destructive payloads are prohibited.
  \item Findings must be reproducible and evidenced; speculation is not graded as a finding.
  \item Communication with the simulated client is via a supplied issue tracker --- responses count toward the professional communication rubric.
\end{itemize}

\subsection*{9.3 Required Methodology}
Learners must follow the curriculum methodology: scope $\to$ map $\to$ enumerate $\to$ probe $\to$ validate $\to$ document. Whitebox source-code analysis is permitted and encouraged.

\subsection*{9.4 Required Findings Report}
Each finding must include: title, severity (with rationale), affected component, reproduction steps, evidence references, business impact, and remediation guidance.

\subsection*{9.5 Required Remediation Recommendations}
For each finding, learners must propose: (a) a code-level fix, (b) a structural or architectural improvement, and (c) a CI/CD or operational guardrail. At least three findings must include verifiable remediation snippets.

\subsection*{9.6 Required Evidence Package}
The submission directory layout uses the following top-level entries: \tool{/findings} for individual finding write-ups, \tool{/evidence} for raw artifacts, \tool{/scope} for the signed ROE and scope diagrams, \tool{/communication} for issue-tracker exports, and \tool{/exec\_summary} for the executive summary. A \tool{HASHES.txt} file at the root must list SHA-256 hashes of every evidence artifact, and a signed \tool{ROE.pdf} is required at the root.

\subsection*{9.7 Required Executive Summary}
A two-page executive summary written for non-technical leadership, with risk in business terms and a prioritized roadmap.

\subsection*{9.8 Grading Rubric}

\begin{tcolorbox}[callout,colback=afpaper,colframe=afnavy,colbacktitle=afnavy,title=Capstone Rubric (100 points)]
\begin{tabularx}{\linewidth}{@{}R{4.6cm} c >{\RaggedRight\arraybackslash}X@{}}
\toprule
\sffamily\bfseries Dimension & \sffamily\bfseries Pts & \sffamily\bfseries Excellent (full credit) means\dots \\
\midrule
Scoping and ROE adherence            & 10 & Scope is signed, faithfully observed, and any deviations are documented and re-authorized.\\
Methodology and evidence             & 20 & A peer can reproduce every finding from the evidence alone.\\
Breadth of findings                  & 15 & At least one client-side, one server-side, one auth/session, and one DB-layer finding.\\
Depth of strongest finding           & 15 & Source-to-sink reasoning, not pattern matching; exploit chain is documented.\\
Remediation quality                  & 15 & Code, structural, and operational layers each present and verifiable.\\
Executive summary                    & 10 & Risk articulated in business terms; roadmap is prioritized and realistic.\\
Professional communication           & 10 & Issue-tracker communication is clear, timely, and respectful.\\

\chapter*{Accessibility, Inclusion, and Teacher Usability}
\addcontentsline{toc}{chapter}{Accessibility, Inclusion, and Teacher Usability}

\section*{11. Accessibility and Inclusion Plan}
The curriculum is designed to be inclusive and accessible across multiple dimensions. The plan addresses skill levels, learning styles, physical accessibility, alternative paths to mastery, collaboration norms, and inclusive examples.

\subsection*{11.1 Different Skill Levels}
\begin{itemize}
  \item \textbf{Foundation track scaffolding:} Each module ships a Module Primer (60--90 minutes of reading) and a guided lab variant.
  \item \textbf{Standard track:} The default lab scenario.
  \item \textbf{Accelerated track:} A ``stretch challenge'' extends each lab with an additional vector or codebase.
\end{itemize}

\subsection*{11.2 Different Learning Styles}
\begin{itemize}
  \item \textbf{Visual:} Architecture diagrams and source-to-sink trace diagrams accompany every module.
  \item \textbf{Auditory:} Live demonstrations and discussion forums are recorded with captions.
  \item \textbf{Kinesthetic:} Every module includes at least one hands-on lab; every advanced module includes an exploit chain.
  \item \textbf{Reading/writing:} Module Primers, findings reports, and reflection journal entries.
\end{itemize}

\subsection*{11.3 Accessible Lab Instructions}
\begin{itemize}
  \item Lab handouts use semantic Markdown with heading hierarchy preserved in PDF exports.
  \item Color is never the sole signal in diagrams; iconography and text labels are paired.
  \item Code blocks specify a language for screen-reader hinting.
  \item Lab tools are tested with keyboard-only navigation; mouse-required steps include a keyboard alternative.
  \item Captions are required on all video; transcripts are required on all audio.
\end{itemize}

\subsection*{11.4 Alternative Paths to Mastery}
Where a learner cannot complete a hands-on demonstration for accessibility reasons, an equivalent demonstration is available: a recorded lab walkthrough with a written analysis, or a paired-programming session with an instructor who manipulates the tooling while the learner directs.

\subsection*{11.5 Collaboration Norms}
\begin{itemize}
  \item A learner-facing Code of Conduct is required reading before Module 1.
  \item Peer review uses a structured rubric to keep critique behavior-focused, not person-focused.
  \item Office hours, written-only Q\&A, and synchronous chat are all available so learners can choose the channel that suits them.
\end{itemize}

\subsection*{11.6 Inclusive Examples and Scenarios}
The curriculum's named scenarios deliberately span sectors (health, civic services, education, retail, public-interest non-profits) and use neutral persona names. Examples avoid culturally narrow references, idioms, or stereotypes. Learners contribute scenario suggestions during the continuous-improvement review.

\section*{12. Teacher Usability Guide}

\subsection*{12.1 Preparing the Lab Environment}
\begin{itemize}
  \item Provision the lab Docker stack from the curriculum repository; verify with the supplied \tool{healthcheck} script.
  \item Apply the network policy (no Internet egress except instructor-controlled collaborator).
  \item Distribute the signed authorization letter and lab charter template at least 48 hours before Module 1.
\end{itemize}

\subsection*{12.2 Pacing}
\begin{itemize}
  \item \textbf{Default 12-week:} 1 module/week; capstone weeks 11--12 (with one pre-capstone consultation each).
  \item \textbf{8-week intensive:} compress Phases A--B; pair Modules 4+5 and 8+9; capstone weeks 7--8.
  \item \textbf{Bootcamp:} 4 weeks at 40~hrs; daily lab review, weekly retro.
\end{itemize}

\subsection*{12.3 Supporting Struggling Learners}
\begin{itemize}
  \item Watch for learners who skip the Module Primer; offer a paired-lab partner.
  \item Review the reflection journal entries weekly; declining specificity is an early warning.
  \item Use the ``walking the trace'' protocol: ask the learner to narrate the source-to-sink path aloud; listen for jumps without justification.
\end{itemize}

\subsection*{12.4 Reviewing Lab Evidence}
\begin{itemize}
  \item Use the supplied evidence-review checklist: layout, hashes, screenshot timestamps, ROE confirmation.
  \item Spot-check by attempting to reproduce one finding from the evidence alone before grading.
\end{itemize}

\subsection*{12.5 Running Discussions Safely}
\begin{itemize}
  \item Open every discussion with a one-sentence ROE reminder and the day's authorized scope.
  \item Redirect ``how do I do this against a real site'' questions immediately and document the redirection.
  \item End discussions with a written summary of decisions and any updates to the lab charter.
\end{itemize}

\subsection*{12.6 Adapting Length}
A four-week ``executive overview'' variant covers Modules 1--3, 4, 6, 9, 11, and a tabletop capstone. A 16-week variant inserts dedicated whitebox practice between Modules 10 and 12, and a second capstone week.

\section*{13. Continuous Improvement Process}
The curriculum is reviewed and improved through a structured process running on three cadences:

\subsection*{13.1 Per-Cohort Review (every cohort)}
\begin{itemize}
  \item Anonymous learner feedback per module (Likert + free-text).
  \item Lab completion data: pass/fail per lab; time-to-completion; retry counts.
  \item Assessment results: per-outcome pass rates; reflection-journal sentiment.
  \item Instructor retrospective with at least two AppSec subject-matter experts.
\end{itemize}

\subsection*{13.2 Quarterly Review}
\begin{itemize}
  \item Web-application-security landscape changes (new framework defaults, browser changes, CVEs of note).
  \item Tooling updates (Burp, ZAP, semgrep, CodeQL, ffuf, nuclei) and their lab implications.
  \item Stakeholder review: training manager, AppSec program owner, hiring-manager input.
\end{itemize}

\subsection*{13.3 Annual Review}
\begin{itemize}
  \item Full alignment-matrix audit against the Minimum Curriculum Quality Checklist.
  \item Review of inclusion plan with at least one accessibility reviewer.
  \item Curriculum-version bump with a public changelog.
\end{itemize}
